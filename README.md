# EnterpriseIQ

### An Intelligent Retrieval-Augmented Generation (RAG) System for Enterprise Knowledge Management

---

## Introduction

EnterpriseIQ is an AI-powered knowledge management system built for large enterprises. The goal of this project is to solve one of the most common problems in large organisations — **information is everywhere, but finding the right answer quickly is nearly impossible**.

Employees across departments waste hours searching through shared drives, emailing colleagues, or digging through outdated document portals just to answer simple questions like:

- *"What is the maternity leave policy?"*
- *"What is the current stock level for SKU-4821?"*
- *"When does our contract with Supplier X expire?"*
- *"What are the steps to raise a procurement request?"*

EnterpriseIQ eliminates this problem by connecting to the organisation's documents and structured data across four core departments — **HR, Finance & Stock, Legal & Compliance, and Operations & Procurement** — and allowing any authorised employee to ask questions in plain natural language and receive accurate, cited answers instantly.

---

## What EnterpriseIQ Will Do Once Completed

Once fully built, EnterpriseIQ will:

- **Ingest enterprise documents** — PDFs and DOCX files from HR, Finance, Legal, and Operations will be automatically parsed, chunked, and indexed into a secure vector knowledge base
- **Sync structured data** — Live inventory records, procurement data, and payroll information will be pulled from the enterprise database and made queryable
- **Answer natural language questions** — Employees will type questions into a chat interface and receive grounded, citation-backed responses generated by a large language model (LLM)
- **Enforce role-based access** — Each employee will only be able to query the departments they are authorised for; a Finance Officer cannot access Legal documents
- **Cite every answer** — Every response will reference the exact source document and page number it was drawn from, eliminating hallucination
- **Log all activity** — Every query, retrieved document, and generated response will be recorded in a tamper-evident audit log for enterprise compliance
- **Provide a management dashboard** — Department managers will be able to upload, review, and delete documents in their namespace through a clean web interface

### Assignment 11 — Persistence Repository Layer

**Abstraction Choice: Factory Pattern**
`RepositoryFactory` selects the storage backend at startup via a `storage_type`
string (`MEMORY`, `FILESYSTEM`, `DATABASE`). No service or domain class changes
when switching backends — only the factory config changes.

| Directory | Description |
|---|---|
| [`repositories/base_repository.py`](./repositories/base_repository.py) | Generic `Repository[T, ID]` interface with 6 CRUD operations |
| [`repositories/interfaces.py`](./repositories/interfaces.py) | 6 entity-specific interfaces with domain query methods |
| [`repositories/inmemory/`](./repositories/inmemory/) | Full HashMap in-memory implementations for all 6 entities |
| [`repositories/filesystem/`](./repositories/filesystem/) | JSON filesystem stubs (Document, AuditLog) |
| [`repositories/database/`](./repositories/database/) | PostgreSQL stubs (UserAccount, Document) |

## CI/CD Pipeline 

### How to run tests locally
```bash
pip install -r requirements.txt
pytest tests/ -v
pytest tests/ --cov=src --cov=services --cov=api --cov-report=term-missing
```

### How the pipeline works

| Trigger | Job | What happens |
|---|---|---|
| Push to any branch | `Run All Tests` | All 215+ tests run automatically |
| PR to `main` | `Run All Tests` | Tests must pass before merge is allowed |
| Merge to `main` | `Build Release Artifact` | Python wheel built and uploaded as artifact |

The `main` branch is protected — no direct pushes allowed. All changes go through a Pull Request that requires passing CI and a code review.
